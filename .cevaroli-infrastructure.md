## Model Expansion

**When new CSV fields arrive:**

1. **Identify the field's purpose** in the CSV documentation or sample data
   - Financial: (e.g., COST_CURRENCY, PROFIT_CENTER) → numeric/enum field
   - Operational: (e.g., WAREHOUSE_ZONE) → text/enum field
   - Temporal: (e.g., LAST_SALE_DATE) → timestamptz field

2. **Determine table placement**
   - If relates to **product master data** (static): `products` table
   - If relates to **commercial data per filial per product**: `product_commercial_data` table
   - If **metadata about imports**: `product_imports` table

3. **Add to Prisma schema** (`/backend/prisma/schema.prisma`):

   ```prisma
   model ProductCommercialData {
     // existing fields...
     newField         String?    // or Int, Decimal, DateTime, etc
     newFieldEnum     Enum?      // if categorical
   }
   ```

4. **Update CSV parser** (`src/lib/services/products.import.ts`):

   ```typescript
   const CSV_COLUMN_MAP = {
     // ... existing mappings
     COST_CURRENCY: "costCurrency", // add new mapping
   };
   ```

5. **Create Prisma migration**:

   ```bash
   cd backend && npx prisma migrate dev --name add_cost_currency
   ```

6. **Add index if needed** (see [Query Optimization](#query-optimization))

7. **Update product_commercial_history snapshot** (JSONB includes all fields)

**Reference**: `/memories/repo/CEVAROLI_INFRASTRUCTURE.md#5-modelo-de-dados`

## Query Optimization

**When encountering slow queries:**

### Step 1: Enable Slow Query Logging

Update `/etc/postgresql/15/main/postgresql.conf`:

```conf
log_min_duration_statement = 500     # log queries > 500ms
```

Restart PostgreSQL and capture slow queries.

### Step 2: Analyze Query Plan

```sql
EXPLAIN ANALYZE
SELECT * FROM product_commercial_data
WHERE abc_curve = 'A'
  AND dias_ate_ruptura < 15
  AND nroempresa = 1;
```

**Look for**:

- Sequential scans on large tables → missing index
- Nested loop joins → consider index on join keys
- Sort operations → consider index on ORDER BY field

### Step 3: Create Appropriate Index

**For Mesa do Comprador (primary workload)**:

```sql
CREATE INDEX idx_pcd_buyer_desk_filters
ON product_commercial_data(nroempresa, abc_curve, dias_ate_ruptura)
WHERE dias_ate_ruptura IS NOT NULL;
```

**For audit/compliance queries**:

```sql
CREATE INDEX idx_audit_created
ON audit_logs(created_at DESC)
WHERE created_at > NOW() - INTERVAL '90 days';
```

**For historical analysis (rare but important)**:

```sql
-- Only if partitioned
CREATE INDEX idx_pch_import_date
ON product_commercial_history(import_id)
WHERE snapshot_data ->> 'abc_curve' = 'A';
```

### Step 4: Validate

```sql
EXPLAIN ANALYZE (same query)  -- should show index usage
SELECT * FROM product_commercial_data WHERE abc_curve = 'A' AND dias_ate_ruptura < 15;
```

## Scaling Decisions

**When data volume increases:**

### Scenario 1: product_commercial_data exceeds 500k rows

**Symptom**: Full table scans even with indexes, VACUUM slowing down imports  
**Decision Point**: Partition by time or by store?

**Recommendation**: **PARTITION BY RANGE (import_date)**

```sql
CREATE TABLE product_commercial_data (
    import_date DATE NOT NULL,
    -- ... other fields
) PARTITION BY RANGE (import_date);

CREATE TABLE pcd_2026_q1 PARTITION OF product_commercial_data
    FOR VALUES FROM ('2026-01-01') TO ('2026-04-01');

CREATE TABLE pcd_2026_q2 PARTITION OF product_commercial_data
    FOR VALUES FROM ('2026-04-01') TO ('2026-07-01');
```

**Why not by nroempresa (store)?** Answer below in [Architectural Rationale](#architectural-rationale).

### Scenario 2: audit_logs exceeds 10M rows

**Symptom**: Audit queries slow, archive needs unclear  
**Decision**: Partition by month + archive strategy

```bash
# Layer 1: Keep last 90 days in PostgreSQL (active partition)
# Layer 2: Archive older than 90 days to S3 Glacier (quarterly restore only)
# Layer 3: Full dump for compliance (yearly)
```

### Scenario 3: MinIO storage exceeds 120GB

**Symptom**: Image uploads slow, disk filling up  
**Decision**: Add second MinIO node or offload to S3

```yaml
# Option A: MinIO distributed mode (adds complexity)
minio server /mnt/ssd1 /mnt/ssd2 /mnt/ssd3 /mnt/ssd4
# Option B: S3 gateway (recommended)
# Redirect hot images to S3, keep 30-day cache locally
```

### When NOT to scale

- **products table** (< 50k rows): Single index sufficient
- **campaigns, slots** (< 10k rows): No partitioning needed
- **promotional_items** (< 1M): Single index on (campaign_id, buyer_id) works
- **users, roles** (< 1k): Cache in memory (Redis)

**Reference**: `/memories/repo/CEVAROLI_INFRASTRUCTURE.md#1-vps--scaling`

## Query Performance Patterns

### Fast Patterns (use these)

✅ **Indexed single-store lookup**:

```sql
SELECT * FROM product_commercial_data
WHERE seqproduto = '12345' AND nroempresa = 1;
-- Uses: idx_pcd_seqproduto + nroempresa index
```

✅ **Filtered batch for buyer desk**:

```sql
SELECT * FROM product_commercial_data
WHERE nroempresa IN (1, 2, 3)
  AND abc_curve IN ('A', 'B')
  AND dias_ate_ruptura BETWEEN 0 AND 30
ORDER BY dias_ate_ruptura ASC
LIMIT 100;
-- Uses: composite index idx_pcd_product_store
```

✅ **Recent audit events**:

```sql
SELECT * FROM audit_logs
WHERE created_at > NOW() - INTERVAL '7 days'
  AND entity_type = 'promotional_item'
ORDER BY created_at DESC
LIMIT 50;
-- Uses: idx_audit_created (partial index on recent data)
```

### Slow Patterns (avoid)

❌ **Full table scan on large table**:

```sql
SELECT COUNT(*) FROM product_commercial_data
WHERE description_erp ILIKE '%xyz%';
-- No index on ILIKE substring search; consider FTS instead
```

❌ **Expensive join without index**:

```sql
SELECT * FROM product_commercial_data pcd
JOIN promotional_items pi ON pcd.product_id = pi.product_id
WHERE pi.campaign_id = 'abc'
-- Missing index on promotional_items(product_id)
```

❌ **Historical query on live table**:

```sql
SELECT * FROM product_commercial_data
WHERE import_date < NOW() - INTERVAL '6 months'
-- Scans entire table; should query product_commercial_history partition instead
```

## Architectural Rationale

### Why partition `product_commercial_history` but not `product_commercial_data`?

**product_commercial_data**:

- Contains **current, operational** state (today's stock, pricing, etc.)
- Queries: "Show me products in store 1 that are about to rupture"
- Size: ~200k rows (steady state across 15k products × 8 filials × 2-3 week imports)
- Access pattern: Sequential access by store + ABC curve
- **Decision**: No partition needed — single index sufficient

**product_commercial_history**:

- Contains **historical snapshots** (one record per product per filial per import date)
- Queries: "Show me stock trend for product XYZ over the last 24 weeks"
- Size: Grows unbounded (121k new rows per import, ~1–2 imports per week = 6–12M rows/year)
- Access pattern: Rare but expensive (analysts, data science)
- **Decision**: Partition by import_date — isolate historical data from operational queries

### Why Redis for cache, not just PostgreSQL?

| Need                                | PostgreSQL                            | Redis                    |
| ----------------------------------- | ------------------------------------- | ------------------------ |
| JWT sessions                        | Possible, but I/O heavy               | ✅ 100µs latency         |
| Query cache (buyer desk filters)    | Possible, but cache invalidation hard | ✅ TTL-based auto-expire |
| BullMQ job queue                    | Overkill for order guarantee          | ✅ Designed for queues   |
| User permissions (cached from auth) | Possible but refresh lag              | ✅ Real-time on login    |

**Rationale**: Redis is 100–1000× faster for cache hits. PostgreSQL is source of truth; Redis is optimized read layer.

### Why MinIO, not just AWS S3?

| Factor       | MinIO               | S3                              |
| ------------ | ------------------- | ------------------------------- |
| Startup cost | ~R$ 0 (local)       | ~R$ 0 (but adds latency/egress) |
| Latency      | < 10ms (local NVMe) | 50–200ms (network)              |
| Egress cost  | None                | ~R$ 0.09 per GB                 |
| Multi-region | Complex             | Built-in                        |
| Compliance   | Full control        | Depends on AWS region           |

**Rationale**: MinIO for hot data (current product images, canvas exports). S3 for cold archive (old backups, > 6 months).

## Common Mistakes & How to Avoid

### Mistake 1: Importing CSV synchronously

❌ **Bad**:

```typescript
app.post("/products/import", (req, res) => {
  const data = fs.readFileSync(req.files[0].path);
  processCsv(data); // Blocks request for 30 seconds
  res.json({ ok: true });
});
```

✅ **Good**:

```typescript
app.post("/products/import", async (req, res) => {
  const import_id = await saveImportRecord();
  await bullQueue.add("csv-import", { import_id });
  res.json({ import_id, status: "processing" });
  // Worker processes in background, notifies via WebSocket
});
```

### Mistake 2: Forgetting to partition before 10M rows

**Cost of late partition** (on live database):

- 2–4 hour maintenance window
- Risk of data loss if interruption
- Can't roll back easily

**Prevention**: Partition `product_commercial_history` from day 1 (even if initially one partition).

### Mistake 3: Storing full CSV snapshots as text, not JSONB

❌ **Bad**:

```sql
snapshot_csv TEXT
```

✅ **Good**:

```sql
snapshot JSONB

-- Then query it:
SELECT snapshot ->> 'abc_curve' FROM product_commercial_history;
```

### Mistake 4: Trusting CSV data without validation

❌ **Bad**: Import everything as-is from the CSV

✅ **Good**:

```typescript
const validate = (row) => {
  if (!["A", "B", "C"].includes(row.abc_curve)) {
    throw new ImportError(`Invalid ABC: ${row.abc_curve}`);
  }
  // ... other validations
};
```

### Mistake 5: Storing images as BLOBs in PostgreSQL

❌ **Bad**: `ALTER TABLE products ADD COLUMN image BYTEA;`

✅ **Good**: Store in MinIO, reference by URL in PostgreSQL.

**Why**:

- PostgreSQL BLOB queries require full table scan
- Image updates lock entire table
- Backups become massive
- CDN caching impossible

## Quick Troubleshooting

### "Import worker stuck/slow"

1. Check if it's reading from MinIO or local disk
2. Verify Redis is not full: `redis-cli INFO memory`
3. Check PostgreSQL connections: `ps aux | grep postgres | wc -l`
4. If stuck: `pm2 restart import-worker`

### "Buyer desk queries slow"

1. Run: `EXPLAIN ANALYZE SELECT ... WHERE nroempresa = 1`
2. Look for Sequential Scan — means missing index
3. Check `/memories/repo/CEVAROLI_INFRASTRUCTURE.md#4-índices` for required indexes

### "Images not loading"

1. Check MinIO is running: `docker ps | grep minio`
2. Check bucket exists: `mc ls play/product-images`
3. Check Nginx proxy config for `img.cevaroli.com.br`
4. Verify firewall isn't blocking 9000: `sudo ufw show`

### "Out of disk space"

1. Check partition sizes: `df -h`
2. Identify culprit: `du -sh /var/lib/postgresql /var/cevaroli/images`
3. Archive old data:

   ```bash
   # Move old images to S3
   mc mirror /var/cevaroli/images/archive hostinger-s3:cevaroli-images-archive

   # Truncate old audit logs
   DELETE FROM audit_logs WHERE created_at < NOW() - INTERVAL '90 days';
   ```
